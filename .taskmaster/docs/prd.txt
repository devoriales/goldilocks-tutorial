# PRD: Goldilocks Kubernetes Tutorial Series

## Overview
Create a complete 10-lesson tutorial series about Fairwinds Goldilocks (v4.15.1) for Kubernetes administrators. The tutorial teaches right-sizing Kubernetes resource requests and limits using Goldilocks, a VPA-based recommendation tool. Structure mirrors the existing Traefik tutorial at ../traefik-application-gateway-tutorial/.

## Target Audience
Kubernetes administrators who are beginners to resource optimization. Comfortable with kubectl and basic Kubernetes concepts but new to VPA and resource tuning.

## Critical Requirements
- Every lesson with shell commands must be VALIDATED on a real k3d cluster before being marked complete
- All content must use new Goldilocks container registry: us-docker.pkg.dev/fairwinds-ops/oss/goldilocks:v4.15.1 (old quay.io path deprecated in v4.15.0)
- Lessons must be short (~10-20 min each)
- Each lesson has: index.html (Docsify), README.md (content), QUIZ.md (4-5 questions)
- Docsify theme color: #f5a623 (golden)

## Task 1: Repository Root Files
Create root-level files: CLAUDE.md (bot guidance), README.md (tutorial overview with lesson index), TUTORIAL_CHECKLIST.md (step-by-step verification), CONTRIBUTING.md, .env.example, and root index.html (Docsify loader). These files establish the project structure and conventions.

## Task 2: Setup Scripts
Create and VALIDATE all scripts in scripts/:
- setup-cluster.sh: Full k3d cluster creation + VPA CRDs + metrics-server install. Cluster name: goldilocks-demo, 1 server + 2 agents.
- verify-prerequisites.sh: Check k3d, kubectl, helm are installed with correct versions
- verify-goldilocks.sh: Verify Goldilocks controller and dashboard pods are Running
- deploy-sample-app.sh: Deploy the metrics-app to the cluster
- cleanup.sh: Delete k3d cluster and clean up
- scripts/README.md: Document all scripts
All scripts must use #!/usr/bin/env bash, set -euo pipefail, color-coded output (GREEN/RED/YELLOW).

## Task 3: Sample Application Manifests
Create sample-app/ with a 3-service "metrics-app" that has intentionally wrong resource settings to generate interesting Goldilocks recommendations:
- namespace.yaml: namespace "metrics-app" with goldilocks.fairwinds.com/enabled=true label
- frontend/deployment.yaml: nginx:stable, intentionally OVER-provisioned (cpu: 500m, memory: 512Mi requests AND limits) - 2 replicas
- frontend/service.yaml: ClusterIP service
- api/deployment.yaml: kennethreitz/httpbin, intentionally UNDER-provisioned (cpu: 10m, memory: 16Mi) - 2 replicas
- api/service.yaml: ClusterIP service
- worker/deployment.yaml: python:3.11-alpine running a CPU-intensive loop, mismatched requests vs limits - 1 replica
- worker/service.yaml: headless service
- load-generator/deployment.yaml: busybox running wget in a loop to generate traffic on frontend and api
- sample-app/README.md: Explain the intentional misconfigurations and why they're educational
VALIDATE: kubectl apply works, pods reach Running state.

## Task 4: Manifests Directory
Create manifests/ with organized Kubernetes YAML:
- 00-vpa/: VPA CRD installation references (link to kubernetes/autoscaler)
- 01-goldilocks/values.yaml: Goldilocks Helm values with new registry, vpa.enabled=false, metrics-server.enabled=false
- 02-recommendations/: Example deployment patches showing before/after resource settings for frontend, api, and worker
- manifests/README.md: Apply order guide

## Task 5: Lesson 01 - The Resource Requests Problem
Create 01-intro-resource-problem/index.html, README.md, QUIZ.md.
Content: Why resource requests/limits matter, OOMKills vs CPU throttling, cost implications of over-provisioning, QoS classes (BestEffort/Burstable/Guaranteed), introduction to the "metrics-app" team narrative. NO cluster required for this lesson.
QUIZ: 4 questions on resource concepts, QoS classes, what happens with no limits.
Duration: ~10 min. Level: Beginner.

## Task 6: Lesson 02 - Local Kubernetes Setup with k3d
Create 02-local-k8s-setup/index.html, README.md, QUIZ.md.
Content: Install k3d, kubectl, helm on macOS/Linux. Create goldilocks-demo cluster. Install VPA from kubernetes/autoscaler GitHub. Install metrics-server. Verify all prerequisites. Reference setup-cluster.sh script.
VALIDATE: Every command in this lesson must work on macOS with k3d. VPA pods must reach Running state.
QUIZ: 4 questions on k3d, VPA purpose, metrics-server role.
Duration: ~20 min. Level: Beginner.

## Task 7: Lesson 03 - VPA Fundamentals
Create 03-vpa-fundamentals/index.html, README.md, QUIZ.md.
Content: What VPA is, VPA modes (Off/Initial/Recreate/Auto), why Goldilocks uses "Off" mode, create a manual VerticalPodAutoscaler object for the api deployment, read VPA recommendations via kubectl describe vpa. Show what raw VPA output looks like before Goldilocks.
VALIDATE: Manual VPA object creation works, recommendations appear after load generator runs for a few minutes.
QUIZ: 4 questions on VPA modes, when to use each, why Off mode is safe.
Duration: ~15 min. Level: Beginner.

## Task 8: Lesson 04 - Installing Goldilocks
Create 04-install-goldilocks/index.html, README.md, QUIZ.md.
Content: Add fairwinds-stable Helm repo. Create goldilocks namespace. Install Goldilocks using helm install with custom values (new us-docker.pkg.dev registry). Verify goldilocks-controller and goldilocks-dashboard pods are Running. Explain architecture: controller watches namespaces, dashboard aggregates VPA data.
VALIDATE: helm install succeeds with new registry path, both pods reach Running.
QUIZ: 4 questions on Goldilocks architecture, what the controller does, Helm chart options.
Duration: ~15 min. Level: Beginner.

## Task 9: Lesson 05 - Enabling Namespaces and Observing VPAs
Create 05-enable-and-observe/index.html, README.md, QUIZ.md.
Content: Deploy metrics-app. Label namespace with goldilocks.fairwinds.com/enabled=true. Watch VPA objects being created automatically by the controller. kubectl get vpa -n metrics-app -w. Show the connection between the namespace label and VPA creation. Inspect a VPA object with kubectl describe.
VALIDATE: VPA objects appear within 60s of labeling the namespace.
QUIZ: 4 questions on namespace labeling, what VPA objects are created, what "Off" mode means for pods.
Duration: ~15 min. Level: Beginner.

## Task 10: Lesson 06 - The Goldilocks Dashboard
Create 06-goldilocks-dashboard/index.html, README.md, QUIZ.md.
Content: Port-forward goldilocks-dashboard service to localhost:8080. Navigate the dashboard. Understand Guaranteed vs Burstable recommendation columns. Explain QoS class implications. Show how to read the recommendation values for frontend (over-provisioned) and api (under-provisioned). Include screenshots described in text (learner should follow along).
VALIDATE: Dashboard accessible at localhost:8080 with recommendations for all 3 services visible.
QUIZ: 4 questions on QoS classes, Guaranteed vs Burstable, how to read dashboard values.
Duration: ~20 min. Level: Beginner.

## Task 11: Lesson 07 - Applying Recommendations
Create 07-applying-recommendations/index.html, README.md, QUIZ.md.
Content: Copy Guaranteed recommendations for the frontend deployment (it's over-provisioned). Copy Burstable recommendations for the api deployment (it's under-provisioned). Update deployment manifests. kubectl apply. Verify pods restart with new resource values. Check QoS class with kubectl get pod -o jsonpath. Run goldilocks summary command for JSON output. Compare before/after resource usage.
VALIDATE: After applying, kubectl describe pod shows new resource values. QoS class is correct.
QUIZ: 5 questions on when to use Guaranteed vs Burstable, what happens to pods when resources change, kubectl patch vs apply.
Duration: ~20 min. Level: Beginner.

## Task 12: Lesson 08 - Advanced Configuration
Create 08-advanced-configuration/index.html, README.md, QUIZ.md.
Content: Per-namespace VPA update mode annotation (goldilocks.fairwinds.com/vpa-update-mode). Container exclusions for sidecars (goldilocks.fairwinds.com/exclude-containers label on deployment). Controller --on-by-default flag for cluster-wide monitoring. VPA resource policy annotations for min/max bounds. Excluding namespaces with --exclude-namespaces.
VALIDATE: Annotations applied, verify changes reflected in VPA objects.
QUIZ: 4 questions on when to exclude containers, what on-by-default does, VPA resource policies.
Duration: ~15 min. Level: Intermediate.

## Task 13: Lesson 09 - Production Considerations

Integrating goldilocks summary JSON into CI/CD pipelines. Cost estimation using recommendations. Related Fairwinds tools (Polaris, Nova, Pluto). When to use VPA Auto mode vs Goldilocks Off mode.
NO cluster validation needed (conceptual lesson).
QUIZ: 4 questions on 8-day convergence, batch workload exclusion, CI/CD integration approach.
Duration: ~15 min. Level: Intermediate.

## Task 15: Killercoda Scenarios
Add goldilocks course to existing repo at /Users/alma/dev-projects/killercoda-scenarios/.
Update killercoda-scenarios/structure.json to add goldilocks entry.
Create goldilocks/structure.json listing 3 scenarios.

Scenario 1 (scenario-1-install-goldilocks): 4 steps
- background.sh: kubectl create namespace goldilocks
- step1: helm repo add fairwinds-stable
- step2: Install VPA CRDs from kubernetes/autoscaler
- step3: helm install goldilocks
- step4: Verify pods Running
- verify.sh: kubectl get pods -n goldilocks --field-selector=status.phase=Running | grep -c goldilocks

Scenario 2 (scenario-2-enable-and-observe): 3 steps
- background.sh: Install goldilocks + deploy metrics-app
- step1: Label the metrics-app namespace
- step2: Watch VPA objects being created
- step3: Inspect a VPA recommendation
- verify.sh: kubectl get vpa -n metrics-app | grep -c frontend

Scenario 3 (scenario-3-apply-recommendations): 4 steps
- background.sh: Full setup with recommendations available
- step1: Read VPA recommendation with kubectl
- step2: Patch the frontend deployment with recommended values
- step3: Verify pod restarts with new resources
- step4: Check QoS class
- verify.sh: kubectl get pods -n metrics-app -l app=frontend -o jsonpath='{.items[0].status.qosClass}' | grep -q Guaranteed

All verify.sh scripts exit 0 on success, exit 1 on failure. backend: kubernetes-kubeadm-1node.
